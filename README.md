# Football Predictor

Two independent layers connected through a shared **PostgreSQL** database hosted on [Neon](https://neon.tech) (free tier).
The app runs on a **Hetzner VPS** (~€4/month). The analyst connects directly to Neon from R — no access to the server needed.

```
football/
├── ingestion/          # pulls data from APIs into the DB
├── modelling/
│   ├── python/         # scikit-learn prediction model
│   └── r/              # your R analysis code goes here
├── shared/             # DB connection config
├── ingestion/main.py   # ingestion entry point
└── modelling/main.py   # modelling entry point
```

## For the analyst (R)

The database is a PostgreSQL instance on [Neon](https://neon.tech) — free, always-on, accessible from anywhere.
No Python setup needed. Connect directly in R and query whatever you need.

### Connecting

Install the R PostgreSQL driver once:
```r
install.packages(c("DBI", "RPostgres"))
```

Then connect using the credentials shared with you:
```r
library(DBI)
library(RPostgres)

con <- dbConnect(
  Postgres(),
  host     = "your-neon-host.neon.tech",   # from Neon dashboard
  port     = 5432,
  dbname   = "football",
  user     = "your_user",
  password = "your_password",
  sslmode  = "require"                      # Neon requires SSL
)
```

### What data is available

#### Match results — `football_matches`

Every match result and upcoming fixture across Premier League, Championship, and Champions League going back to the 2020/21 season.

| Column | Description |
|---|---|
| `match_id` | Unique match ID |
| `utc_date` | Kick-off time (UTC) |
| `status` | `FINISHED`, `SCHEDULED`, or `POSTPONED` |
| `competition_id` | 2021 = Premier League, 2016 = Championship, 2001 = Champions League |
| `competition_name` | Full competition name |
| `home_team_id` | Home team ID (links to `FootballTeams`) |
| `home_team` | Home team name |
| `away_team_id` | Away team ID |
| `away_team` | Away team name |
| `winner` | `HOME_TEAM`, `DRAW`, `AWAY_TEAM`, or `null` if not played |
| `home_goals` | Full-time home goals (`null` if not played) |
| `away_goals` | Full-time away goals (`null` if not played) |
| `last_updated` | Timestamp of last API update |

```r
# All finished Premier League matches
pl_matches <- dbGetQuery(con, "
  SELECT *
  FROM football_matches
  WHERE competition_id = 2021
    AND status = 'FINISHED'
  ORDER BY utc_date
")
```

#### Python model predictions — `football_predictions`

Outcome probabilities generated by the Python model for upcoming fixtures. Available after `python -m modelling.main predict` has been run.

| Column | Description |
|---|---|
| `match_id` | Links to `football_matches` |
| `utc_date` | Kick-off time |
| `competition` | Competition name |
| `home_team` | |
| `away_team` | |
| `prob_home_win` | Model probability (0–1) |
| `prob_draw` | Model probability (0–1) |
| `prob_away_win` | Model probability (0–1) |
| `predicted_outcome` | `Home Win`, `Draw`, or `Away Win` |
| `predicted_on` | Date prediction was made |

```r
# Compare Python model predictions vs actual results
comparison <- dbGetQuery(con, "
  SELECT
    p.home_team,
    p.away_team,
    p.predicted_outcome,
    p.prob_home_win,
    p.prob_draw,
    p.prob_away_win,
    m.winner        AS actual_winner,
    m.home_goals,
    m.away_goals,
    p.predicted_on
  FROM football_predictions p
  JOIN football_matches m ON p.match_id = m.match_id
  WHERE m.status = 'FINISHED'
  ORDER BY p.utc_date
")
```

---

#### Model accuracy log — `football_prediction_accuracy`

Running record of how accurate the Python model has been over time, broken down by outcome type.

| Column | Description |
|---|---|
| `evaluated_on` | Date the accuracy check was run |
| `outcome` | `Home Win`, `Draw`, or `Away Win` |
| `correct` | Count of correct predictions for this outcome |
| `total` | Total predictions for this outcome |
| `accuracy` | `correct / total` |
| `overall_accuracy` | Accuracy across all outcomes on that date |

### Useful joins

```r
# Full match results with team details
full_matches <- dbGetQuery(con, "
  SELECT
    m.utc_date,
    m.competition_name,
    h.name AS home_team, h.venue AS home_venue,
    a.name AS away_team,
    m.home_goals, m.away_goals, m.winner
  FROM football_matches m
  JOIN football_teams h ON m.home_team_id = h.id
  JOIN football_teams a ON m.away_team_id = a.id
  WHERE m.status = 'FINISHED'
")

# Home and away goal averages per team per season
goal_avgs <- dbGetQuery(con, "
  SELECT
    home_team_id AS team_id,
    home_team,
    EXTRACT(YEAR FROM utc_date) AS year,
    AVG(home_goals::float) AS avg_home_gf,
    AVG(away_goals::float) AS avg_home_ga
  FROM football_matches
  WHERE status = 'FINISHED'
  GROUP BY home_team_id, home_team, EXTRACT(YEAR FROM utc_date)
")
```

Place your `.R` scripts in `modelling/r/`. Full column-level detail is in [`modelling/r/schema.md`](modelling/r/schema.md).

## For the engineer (Python)

### Infrastructure (Hetzner + Neon)

**Database — [Neon](https://neon.tech)** (free forever)
1. Create a free account at neon.tech
2. Create a project → database named `football`
3. Copy the connection string from the dashboard
4. Share read-only credentials with your analyst

**Server — [Hetzner](https://hetzner.com) CX22** (~€4/month)

Deployment is automated via GitHub Actions — every push to `main` builds a Docker image, pushes it to GitHub Container Registry, and SSH-deploys to the server.

First-time server setup:
```bash
git clone <your-repo> /app/football
cd /app/football
bash scripts/server_setup.sh
```

The setup script installs Docker, writes cron jobs, and creates `/app/football/.env` from the template. Edit it with your credentials:
```bash
nano /app/football/.env
```

**Cron jobs (installed automatically by setup script)**

| Schedule | Command |
|---|---|
| Daily 23:30 | `ingestion.main today` — pull today's results |
| Monday 02:00 | `ingestion.main standings` — refresh league tables |
| Monday 03:00 | `modelling.main predict` — predict next 7 days |
| Daily 23:45 | `modelling.main accuracy` — score settled predictions |

**CI/CD — GitHub Actions**

Add these secrets to your GitHub repository (`Settings → Secrets → Actions`):

| Secret | Value |
|---|---|
| `HETZNER_HOST` | Your server IP |
| `HETZNER_USER` | SSH username (e.g. `root`) |
| `HETZNER_SSH_KEY` | Your private SSH key (contents of `~/.ssh/id_rsa`) |

Every push to `main` will build the image and deploy automatically.

**Cost: ~€4/month total.** The database is free on Neon's free tier (0.5GB, always on).

### Ingestion

```bash
python -m ingestion.today                    # today's results
python -m ingestion.backfill                 # 2020 → now
python -m ingestion.backfill 2022 2023 2024  # specific seasons
python -m ingestion.standings                # league tables
python -m ingestion.standings 2023 2024      # specific seasons
python -m ingestion.teams                    # team reference data
```

### Modelling

```bash
python -m modelling.main train            # train default model
python -m modelling.main train --compare  # grid search across 4 model types
python -m modelling.main predict          # predict upcoming fixtures
python -m modelling.main predict --retrain
python -m modelling.main accuracy         # score settled predictions
```

### Adding a new data source

1. Create `ingestion/sources/<source_name>/`
2. Write to the same DB table schemas
3. Register in `ingestion/main.py`

The modelling layer needs no changes — it only reads from the DB.
